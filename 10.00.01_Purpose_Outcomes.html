GroovConnect OAS™ · Module 10, Section 0 · Instructor Orientation

Purpose & outcomes — Operational Intelligence & Flywheel Mastery

The Analyst-to-Architect transition. From fixing broken systems to designing systems that fix themselves.

Analyst mindset
"What should we fix?" → A solution to the current problem. Requires a follow-up engagement when the next problem appears.
→
Architect mindset
"What system prevents this permanently and improves over time?" → A feedback loop that catches the problem earlier with each cycle — eventually before the operator notices it.
Linear model
Each lesson is a service delivered. More staff effort produces more performance — until capacity is reached. Performance plateaus.
→
Compounding model
Each lesson is data harvested. Better data produces better predictions. Better predictions produce better outcomes. No structural ceiling.
One school's gain
One school's experience benefits that school alone.
→
Network intelligence
Every school's data improves outcomes for all schools in the network. Each new school makes every existing school more accurate.
MSOI calculation
Two consecutive scenarios within ±2 points. All five component formulas recited without reference.
Flywheel explanation
90-second partner drill — partner can repeat the core loop without prompting.
Flywheel design
≥5 nodes, each with a named data output. Closed loop. Passes the Autonomy Test.
Autonomy Test
"If you leave for 6 months, does this get better?" → Answer is "better," with a specific feedback loop named.
The Autonomy Test is the single most important diagnostic in this module. If a trainee cannot say "it gets better" and explain specifically which data feeds back and which output quality rises — they are designing Structured systems while talking about Intelligent ones. The distinction is the feedback loop — not the ambition.
